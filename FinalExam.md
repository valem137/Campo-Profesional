## 2. GIT, VERSIONAMIENTO Y TRABAJO PROFESIONAL 

### 4. Explica con tus palabras el flujo básico de trabajo en Git (commit, push, pull, branch, merge).

Luego de realizar cambios en un proyecto, se guardan con un `commit`, que es como tomar una “foto” del estado del código al momento en que se dejó.

Luego se hace `push` para enviar esos cambios al repositorio remoto. Si otro colaborador hizo modificaciones, se hace `pull` para traerlas a mi equipo. 
+
























Si se quiere trabajar en una nueva función sin afectar el código principal, se crea una `branch`. Cuando termina esa función, uno mis cambios a la rama principal utilizando `merge`.

### 5. ¿Por qué es importante usar control de versiones en proyectos reales?

Permite trabajar en equipo sin perder los avances, evita sobrescribir archivos, también ayuda a rastrear errores y mantiene un historial completo del proyecto. También hace que el trabajo sea más profesional, seguro y organizado.

### 6. ¿Cómo aplicarías GitHub en un proyecto profesional o universitario en equipo?

Se crea un repositorio central. Luego, para evitar problemas en el código principal, cada persona crea una rama separada para hacer sus cambios. Una vez que terminan, guardan sus avances con un commit y los envían al repositorio en línea con push. Cuando el trabajo está listo y revisado, se usa un Pull Request para pedir que se una esa rama a la rama principal usando merge, integrando así los cambios de forma segura.
